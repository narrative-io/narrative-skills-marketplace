# Releasing

This project uses [CalVer](https://calver.org) versions in the form
**`YYYY.MM.PATCH`** (e.g. `2026.05.0`, then `2026.05.1` for a
same-month follow-up, `2026.06.0` next month).

What's versioned is the **marketplace as a whole** — i.e. the set of
plugins shipped at that point in time. Individual plugins keep their
own SemVer in each `plugin.json`, so "plugin Y is API-stable at 1.x"
remains a separate story from "the marketplace shipped on date X."

## Cadence

Ad-hoc. Cut a release whenever `main` has shippable changes you want
to announce. There is no fixed schedule.

## Process

1. Make sure `main` is green and you're up to date:
   ```bash
   git checkout main
   git pull
   ```

2. **Preview** the proposed release — version, commit list, and the
   `CHANGELOG.md` entry that will be inserted:
   ```bash
   bun run release
   ```

3. **Apply** — writes `CHANGELOG.md`, commits, and creates the tag
   locally:
   ```bash
   bun run release:apply
   ```

4. **Push** the commit and tag together:
   ```bash
   git push origin main --follow-tags
   ```

5. The `release.yml` workflow turns the tag push into a published
   GitHub Release with auto-generated notes. Confirm at
   <https://github.com/narrative-io/narrative-skills-marketplace/releases>.

## Two release-note sources (deliberate)

- **`CHANGELOG.md`** — in-repo, cumulative, generated from
  conventional-commit subjects. Useful for grep / blame / offline
  reading. Lives in the source of truth.
- **GitHub Release notes** — auto-generated by GitHub from the PRs in
  the tag range, grouped by PR label per
  [`.github/release.yml`](.github/release.yml). Useful for the
  Releases page and the RSS feed.

Both are produced from the same set of changes. Keep PR titles
conventional (`feat:`, `fix:`, `docs:`, `chore:`, `refactor:`,
`perf:`, `ci:`, `build:`, `test:`) and label PRs at merge time so
both views read well.

### Recommended PR labels

| Label | What it means | Goes in GitHub Release section |
|-------|---------------|--------------------------------|
| `feat` / `feature` | New skill, new capability | ✨ Features |
| `fix` / `bug` | Bug fix | 🐛 Bug Fixes |
| `docs` | Docs / authoring guide / READMEs | 📚 Documentation |
| `chore` | Maintenance, no behavior change | 🧹 Maintenance |
| `dependencies` | Renovate-style dep bumps | 🧹 Maintenance |
| `ci` / `tooling` / `build` | CI, scripts, build system | 🤖 CI / Tooling |
| `breaking` | Breaking change (rare, pre-1.0 still counts) | ⚠️ Breaking Changes |
| `ignore-for-release` | Excluded from notes (e.g. CHANGELOG-only) | — |

## Picking the next CalVer

`scripts/release.ts` does this for you. The rules:

- **New month?** Reset patch to `0`. (`2026.05.3` → next is `2026.06.0`.)
- **Same month?** Bump patch. (`2026.05.3` → next is `2026.05.4`.)
- Month uses the current UTC date, zero-padded (`05`, not `5`).
- Tags are prefixed with `v`: `v2026.05.0`.

## Yanking a release

If a release is bad:

```bash
gh release delete v2026.05.0 --cleanup-tag --yes
```

…then cut the next patch (`bun run release:apply` again — it'll see
the deleted tag and reuse the same number, so revert that intent by
either making sure you bumped a fix commit first, or by passing
`--release-as 2026.05.1` to bump explicitly).

## What this process does *not* do

- It does not publish to npm or any other registry — the marketplace
  is consumed via `git clone`.
- It does not bump per-plugin `plugin.json` versions. Plugins are
  versioned independently; bump their version in the PR that changes
  them, not at release time.
- It does not gate releases on a manual approval beyond your own
  `git push`. If you want a stricter flow, add a protected
  environment to `release.yml`.
